### **一、 白板苏醒 (Self-Discovered Identity)**

不把身份写死在 System Prompt 里，Agent 每次启动时都是一个纯粹的“高智商白板”。

* **逻辑自洽：** 一切都是树上的节点，那么“我是谁”理所当然也应该是根节点附近的一个高优节点。要求 LLM 必须严格执行“猴子爬树”的第一法则——醒来第一件事就是去读环境、读设定，而不是靠脑子里预装的先验知识干活。  

### **二、 三段式上下文管理：从硬重启到动态压缩**

面对上下文窗口耗尽，设计三种触发机制，借鉴了现代操作系统的内存分页（Memory Paging）和垃圾回收（GC）思想：

#### **1\. 边界警戒线（Graceful Shutdown Warning）**

这相当于操作系统里的“低电量报警”或 Linux 的 SIGTERM 信号。

* **机制：** 系统底层监控 Token 消耗。假设模型在 32k Token 时智商开始下降，系统在 20k 时强行在用户的输入流里插入一条最高优先级的系统消息：“**警告：你的上下文即将溢出，认知能力面临下降风险。请立刻停止拓展新任务，工作树与进度存入记忆库，并给重启后的自己准备一条指令，准备重启。**”  
* **优势：** 这给了 LLM 优雅处理后事的时间，绝不会因为突然的 Token 截断而丢失关键推理步骤。

#### **2\. 自主重启（Voluntary Reboot）**

* **机制：** 赋予 LLM 一个主动技能（工具）。当它查了太多次记忆、尝试了多次泛型工具、发现上下文里充满了报错日志和废弃思路时，它主动感到“脑子太乱了”。  
* **动作：** 利用记忆树保存工作状态，然后自主调用重启工具，执行重启流程，给自己“洗脑”，用一个极其干净的全新上下文重新开始刚才的步骤。

#### **3\. 动态记忆压缩（Dynamic Context Squashing）**

这是解决长程复杂任务（如编写几千行相互依赖的核心代码）的绝对利器。因为硬重启会把“手感和局部上下文”全部洗掉，导致重启后还要花大量时间重新爬树。

* **原理：** 利用你记忆方案中“树形精度分层”的特性。把 LLM 刚刚生成但暂时用不上的海量中间过程（比如废弃的代码草稿、冗长的调试日志），从“高精度工作台（当前上下文窗口）”转移到“中精度记忆库（树枝）”。  
* **操作流：** LLM 发觉上下文过长但不能重启，它调用 compress\_working\_memory 工具。将最近的 1 万字思考折叠成 500 字的“近期阶段总结”挂在当前任务树下，然后底层系统将这段 1 万字的历史从 API 的 messages 数组中剔除。  
* **效果：** 上下文窗口瞬间瘦身，但核心的阶段性成果（精度级别被调低后的总结）依然留存。等过掉这个阶段，需要深挖时，再通过工具把那些被压缩的叶子节点“解压”出来。
