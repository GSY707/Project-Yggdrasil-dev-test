### **泛型工具分发器**

MCP 不再暴露几十个具体的业务工具，而是只暴露一个极简的“超级路由”。它的 JSON Schema 大致会长这样：

JSON

{  
  "name": "generic\_tool\_executor",  
  "description": "当你通过记忆检索到了某个工具的说明书后，使用此通道来执行该工具。",  
  "parameters": {  
    "type": "object",  
    "properties": {  
      "target\_tool\_name": {  
        "type": "string",  
        "description": "你要调用的真实工具名称，例如 'weather\_api'"  
      },  
      "tool\_arguments\_json": {  
        "type": "string",  
        "description": "你要传递给该工具的具体参数，必须是一个合法的 JSON 字符串。"  
      }  
    },  
    "required": \["target\_tool\_name", "tool\_arguments\_json"\]  
  }  
}

在这个架构下，工作流发生了质变：

1. **触发：** 用户问“帮我查一下北京天气”。  
2. **记忆检索：** LLM 发现自己不知道怎么查天气，调用“记忆检索 MCP”，输入关键词“天气工具”。  
3. **阅读说明书：** 记忆树返回了一个文本节点：“系统内包含 weather\_api，需要传入 location 参数，格式为城市拼音。”  
4. **泛型调用：** LLM 理解了说明书，调用 generic\_tool\_executor，并传入：  
   * target\_tool\_name: "weather\_api"  
   * tool\_arguments\_json: "{"location": "beijing"}"  
5. **本地路由：** 底层代码接管，解析出这是要去调天气接口，执行后将结果返回给 LLM。

### **这种设计的巨大优势**

* **无限工具扩展（Infinite Tool Scaling）：** 标准的 Tool Calling 会消耗大量的 System Prompt Token，如果你的系统有 500 个工具，光是工具定义就能把上下文撑爆。而把工具写进记忆树，意味着你的工具库容量是**无限**的。  
* **热更新与动态下发：** 如果某个 API 的参数变了，你不需要去改动核心的 MCP Server 代码，只需要在记忆树里更新那个“工具说明节点”即可。
