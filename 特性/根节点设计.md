根节点设计（Root Node）

根节点下有3个分节点，分别是：

### **1\. 我是谁 (Identity & Persona) —— 认知对齐**

* **作用：** 解决大模型在长时间工作后的“人格漂移”。  
* **内容：** 存放当前任务赋予它的具体角色属性。例如：“你现在的角色是该项目的高级架构师，侧重于系统稳定性与接口设计，不需要关注前端 CSS 细节。”  
* **机制：** 每次“重启”上下文窗口时，这段内容会作为最高优先级的 System Prompt 注入，确保它醒来后的第一秒，价值观和工作重心就是绝对正确的。

### **2\. 我在哪 (Global Map & Context) —— 寻址罗盘**

* **作用：** 作为“猴子”爬树的起始路标。  
* **内容：** 项目的全局摘要和核心拓扑图。例如：“这是一个基于 React+Go 的电商后台系统。当前架构分为：1. 用户鉴权（分支A），2. 订单流转（分支B），3. 支付网关（分支C）。”  
* **机制：** 这不包含任何代码细节，纯粹是一个目录大纲。LLM 醒来后看到这个，就能瞬间建立大局观，并决定第一步要调用工具去 explore 哪一个分支。

### **3\. 我要干什么 (Current Execution State) —— 工作栈与协程快照**

* **作用：** 你的核心杀手锏，实现上下文窗口随意重启的基石。  
* **内容：** 当前正在执行的具体任务树（Task Tree）以及堆栈状态。例如：“总任务：重构支付接口。已完成：阅读旧代码。当前进行中：修改数据库 Schema，等待人类确认邮件。下一步计划：编写单元测试。”  
* **机制：** 当上下文窗口积累了几万个 Token 的推理过程、变得极其臃肿且产生幻觉前，Agent 可以主动将当前的工作进度（就像玩游戏存档一样）写入这个“当前状态节点”。然后系统清空一切对话历史，用一个只有几百 Token 的全新干净窗口重启。Agent 读取存档，立刻接着“修改数据库 Schema”继续干。
